## Numeric Encoding Status

The main unsolved piece is RADAN's compact numeric encoding in DDC field data.

Examples from `B-28`:

- a DXF line from `(1.75, 1.322835)` to `(0.25, 1.322835)` becomes a `G` record with compact geometry data:
  - `o?<.0@6aR?@_j2@.o?X..............`
- after Y normalization, the likely numeric sequence is:

## Current Encoding Hypotheses

What seems solid:

- DDC geometry record order equals DXF modelspace order for the F54410 paint pack.
- `G` records are lines.
- `H` records are arcs and circles.
- record field index `8` is pen.
- record field index `10` contains compact geometry data.
- empty dot slots often represent zero, unchanged, omitted, or default values.
- RADAN normalizes part geometry into positive local symbol coordinates.
- `G` line geometry often decodes like normalized start `X`, normalized start `Y`, delta `X`, and delta `Y` in slots `0..3`, but that is not enough to produce a RADAN-valid closed part.
- for some line delta slots, empty token decodes as zero delta.
- `H` arc geometry may decode like start point, end delta, center delta, and constant flags, but angle/sweep/orientation semantics are not proven.
- `H` circle geometry often looks like the same shape with no end delta:
  - slot `0`: center `X + radius`
  - slot `1`: center `Y`
  - slot `4`: `-radius`
  - slots `6` and `9`: constant `1`

What is not solved:

- how RADAN interprets DDC topology/closure beyond decoded endpoints
- whether line/arc records have angle, sweep, orientation, or state semantics that must be preserved exactly
- whether `encode_ddc_number(...)` needs to mimic RADAN's final continuation-digit choices exactly, or whether any valid equivalent token is accepted by RADAN
- how to regenerate all derived symbol metadata safely enough for production

Most likely approach:

- solve the DDC field format from real generated symbols first, including closure/topology and arc angle semantics
- use COM scan only as an oracle, not as the production converter
- generate a native `.sym` prototype for one small known part
- validate by adding the prototype `.sym` to a copied `.rpd` and opening/nesting in RADAN; file-level decoded geometry checks alone are not acceptable

## Overnight Lab Update: Exact Endpoint Canonicalization

The `B-14` failure was not only an angle/line-formula problem. `B-14` is line-only, and the synthetic file had decoded floating point values that looked correct, but shared endpoints did not always decode to the same exact DDC fraction.

Example failure mode:

- one record encoded a point as an absolute start coordinate
- the adjacent record reached the same displayed point as `start + delta`
- both values displayed as the same 6-decimal coordinate
- the exact decoded fractions differed by around `1e-17`

That is enough for our exact contour check to see an open endpoint, and it is a plausible explanation for RADAN showing missing/broken line geometry.

Implemented lab fix:

- `ddc_number_codec.py`
  - added `encode_ddc_number_fraction(...)`
  - kept the normal float encoder behavior for existing callers
- `write_native_sym_prototype.py`
  - added `--canonicalize-endpoints`
  - added `--topology-snap-endpoints`
  - canonical endpoint values are encoded once to DDC fractions
  - line/arc deltas are then encoded as the exact difference between those encoded endpoint fractions
  - this makes `start + delta` land exactly on the same DDC fraction as the adjacent endpoint
- `compare_ddc_geometry.py`
  - new comparator for decoded slot-level deltas between RADAN-direct oracle SYMs and synthetic SYMs

Important correction:

- source coordinate rounding should round source points and the source min bounds to 6 decimals
- it should not round radii
- circle radii such as `0.1015625` are real inch fractions and RADAN-direct preserves them

Two distinct failure classes are now evident:

1. Arc math/rounding loss:
   - `B-28`, `B-30`, and `F54410-B-41` have real differences in `H`/`ARC` geometry slots.
   - The current synthetic writer rounds shared source endpoints to 6 decimals before encoding.
   - RADAN-direct templates preserve more exact arc-derived values such as:
     - `B-28`: `0.234923155196` vs synthetic `0.234923`
     - `F54410-B-41`: `0.044194173824` / `0.018305826176` vs synthetic `0.044194` / `0.018306`
     - `B-30`: `0.025114741004` vs synthetic `0.025115`
   - This points at non-cardinal arc endpoint/center-delta generation. The writer should stop flattening arc trig values to 6-decimal source points, or should preserve template tokens when the existing template already matches the source part.

2. Token representation sensitivity:
   - `B-17`, `B-27`, and `F54410-B-49` decode essentially identical to their templates and have no odd exact endpoints in the synthetic output.
   - Despite that, RADAN/user-visible geometry is bad.
   - Example line-only `F54410-B-49` record 17:
     - template: `3@>YOSdL\`@6..3@^8[jn5U2X...`
     - synthetic: `3@>YOSdL\`@3..3@^8[jn5U2Sb...`
     - both decode to the same geometry within about `1e-14`
   - This is strong evidence that RADAN's DDC token byte representation is not freely interchangeable, or that some derived/cache metadata is tied to the exact encoded geometry tokens.

Immediate implication:

- decoded numeric comparison and exact endpoint closure are necessary but not sufficient
- a promotion gate needs either RADAN/user-visible validation, exact template-token preservation for already-existing templates, or a more exact encoder that reproduces RADAN's token choices
- the current synthetic writer should not be trusted for production geometry regeneration

Useful finding:

- RADAN sometimes keeps an explicit final `0` continuation digit in a DDC number token.
- Our exact fraction encoder was previously only able to emit the shorter equivalent token.
- `ddc_number_codec.encode_ddc_number_fraction(...)` now accepts `min_continuation_digits` for lab experiments that need byte-shape preservation.

Limit of that finding:

- The precision-shape candidate improves the whole corpus by `516` token slots and `26` exact records.
- It does not improve `F54410-B-49` at all.
- B-49 remains `0 / 28` exact geometry records and `407 / 476` matching token slots for the best generative candidates.

Most important blocker:

- B-49 is line-only, so this is not an arc formula problem.
- Fresh RADAN and the 10:27 backup generate byte-identical DDC geometry for B-49.
- Our decoded values are still within around `1e-14`, but all B-49 geometry records differ at the token byte level.
- The same rounded/displayed value can map to multiple oracle tokens across the corpus. That means a 6-decimal numeric match is not enough to reproduce RADAN's accepted token stream.

Current assessment:

- The synthetic writer is not production-ready.
- The remaining problem is RADAN's internal value-selection and serialization behavior.
- Public RADAN documentation confirms RADAN/Radimport owns DXF conversion, geometry healing, and `.sym` creation workflows, but no public DDC geometry-token specification was found in the searched material.
- Without that value-selection rule, direct no-RADAN DXF-to-SYM generation is still a research project, not a dependable operations path.

Next research gate:

Create a tiny controlled DXF corpus and convert it once through RADAN with all visible RADAN sessions closed. Vary one coordinate at a time and compare tokens for:

- exact dyadic values: `0.25`, `0.5`, `2.5`, `20.625`
- common decimal values: `0.025`, `0.05`, `0.95`
- B-49 canary values: `0.510833`, `0.512607`, `6.299218`, `10.098437`, `30.64833`

If those one-variable oracles do not produce a deterministic value-selection rule, the hack should be parked and production should stay on RADAN conversion with a hard preflight that requires user RADAN sessions to be closed.

## 2026-04-28 Preliminary DXF Cleaning Experiment

Added:

- `clean_dxf_outer_profile.py`
- `tests/test_clean_dxf_outer_profile.py`

Rationale:

`F54410-B-49` is not topologically corrupt, but its outside profile contains eight tiny stair-step segments around bend/relief-looking corners. Each tiny segment is about `0.001774174 in`. These features look like flat-pattern distortion artifacts and are exactly the kind of geometry that can make byte-faithful DDC reproduction fragile.

The cleaner uses `ezdxf.edgesmith`/`ezdxf.edgeminer` to find closed loops on the profile layer, selects the largest loop as the outside profile, and applies a local bounded vertex-removal simplifier. It writes a cleaned DXF copy plus a JSON report. It does not modify the source DXF.

Hard path rule:

- `W:` is read-only source truth for this workflow.
- Cleaned DXFs, reports, synthetic SYMs, project edits, and logs must not be written to `W:`.
- The only allowed `W:` mutation is the Inventor-owned generated handoff pair: `*_Radan.csv` and `*_report.txt`, which are moved to the matching `L:` project folder.

Current v1 limits:

- LINE-only outside loops only
- holes/interior loops are not rewritten
- arcs/splines are reported and skipped until a curve-aware cleanup pass exists
