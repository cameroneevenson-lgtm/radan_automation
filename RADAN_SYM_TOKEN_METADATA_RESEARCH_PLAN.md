# RADAN SYM Token and Metadata Research Plan

Last updated: 2026-04-29

## Purpose

Crack why RADAN-created `.sym` files work visually and operationally when decoded geometry alone is not enough.

This plan is not primarily about cleaned DXFs. Cleaned DXFs may remain useful as a separate production input option, but this research pass is focused on RADAN's native token choices, file metadata, and any derived cache or wrapper state that makes a symbol trustworthy in RADAN.

Primary question:

- Given a source DXF and a known-good RADAN-created `.sym`, can we reproduce a native-looking `.sym` without asking RADAN to convert the part during normal import?

Secondary questions:

- Which bytes/records must match RADAN exactly?
- Which metadata/cache fields are ignored, derived, or display-critical?
- Can a per-part or per-family RADAN-created donor safely provide wrapper/cache state while we rewrite only a proven subset?
- Is there a hidden/headless RADAN call that rebuilds metadata/cache without disturbing open user sessions?

## Current Evidence

Known facts:

- DXF entity order has matched DDC geometry record order for the F54410 paint-pack corpus.
- DXF `LINE` maps to DDC `G`.
- DXF `ARC` and `CIRCLE` map to DDC `H`.
- DDC field index `8` is the pen field.
- DDC field index `10` contains compact geometry data.
- `IV_INTERIOR_PROFILES` maps to pen `1`.
- `IV_MARK_SURFACE` maps to pen `7`.
- Decoded geometry validation can pass while RADAN still shows visibly bad symbols.
- `B-17`, `B-27`, and `F54410-B-49` previously decoded close to good templates but still had visible failures.
- `B-28`, `B-30`, and `F54410-B-41` exposed real arc/endpoint value loss from the current writer.
- Universal donor mode failed. Do not use it operationally.
- Per-part RADAN-created symbols are valuable as good donors/oracles.

Most important correction:

- A file-level decoded geometry pass is not sufficient. The research target is exact RADAN behavior, especially token choice and metadata/cache sensitivity.

## Answer: Do We Need RADAN Proper?

Not at the start.

We can do a large amount of useful work offline from existing good `.sym` donors:

- catalog known-good symbols
- diff wrappers, attributes, DDC blocks, bounds, and unknown fields
- compare good RADAN symbols against our synthetic failures
- infer token patterns from the existing corpus
- build mutation matrices and lab variants

But yes, to actually crack RADAN's token choices and cache/metadata behavior with confidence, we probably need limited controlled access to RADAN proper.

Reason:

- The existing production corpus has too many variables changing at once.
- RADAN appears to choose different compact tokens for values that decode to the same displayed number.
- To isolate that rule, we need tiny one-variable oracle files generated by RADAN: one line, one arc, one circle, one coordinate change at a time.
- To determine cache sensitivity, we need RADAN to load or regenerate deliberately mutated lab symbols and tell us whether it displays them correctly.

Recommended RADAN access model:

- no production files
- no open user RADAN sessions
- one lab folder
- one or a few tiny DXFs per run
- kill/verify RADAN processes before and after
- log every generated oracle

That should consume a seat only during the controlled oracle run, not during normal import or offline analysis.

## Phase 0: Setup and Donor Inventory

Goal:

- build a trusted index of all usable RADAN-created donor/oracle symbols.

Inputs:

- L-side project symbols that the user has confirmed are good
- `_headless_import_backups` folders
- lab folders explicitly named `radan_known_good` or `radan_oracle`

Exclude:

- synthetic output folders
- universal donor outputs
- any symbol with `Attr 110 = donor`
- partial failed runs unless explicitly labeled for negative-control comparison

Deliverables:

- `build_sym_oracle_index.py`
- `sym_oracle_index.json`
- `sym_oracle_index.csv`

For each symbol record:

- part name
- path
- source classification: production-good, backup-good, lab-oracle, synthetic, donor, unknown
- file size and modified timestamp
- DDC record count and type sequence
- pen distribution
- bounds/extents extracted from all known locations
- wrapper/header fingerprint
- DDC geometry fingerprint
- unknown-field fingerprint

Pass gate:

- every symbol used as an oracle has a provenance label
- synthetic and universal-donor symbols are excluded from normal template selection
- duplicates are grouped by part and fingerprint

## Phase 1: Good vs Failed Section Diff

Goal:

- discover whether visible failures follow the DDC block, wrapper metadata, cache fields, or a combination.

Target parts:

- `B-17`
- `B-27`
- `B-28`
- `B-30`
- `F54410-B-41`
- `F54410-B-49`
- `B-14`
- one simple exact-match part such as `B-10`
- one close-match part such as `B-16`

Deliverables:

- `sym_section_diff.py`
- `sym_hybrid_matrix.py`
- `section_diff_<part>.json`
- `hybrid_matrix_<part>.json`

Work:

1. Parse known-good RADAN symbol.
2. Parse current synthetic symbol, if available.
3. Split the file into stable sections:
   - XML/document wrapper
   - attributes
   - DDC geometry records
   - non-DDC symbol data
   - bounds/extents fields
   - preview/cache-like fields
   - unknown binary/text payloads
4. Create lab-only hybrid variants:
   - good wrapper + synthetic DDC
   - synthetic wrapper + good DDC
   - good wrapper + synthetic DDC + good bounds
   - good wrapper + good DDC + synthetic bounds
   - good wrapper + synthetic DDC + regenerated known fields
5. Compare each hybrid with file-level validators.
6. If RADAN access is approved, open only the smallest hybrid set in RADAN and record visual status.

Expected result:

- identify whether failures are purely token/DDC, purely metadata/cache, or mixed.

Pass gate:

- for each known-bad canary, classify the failing section with evidence.

## Phase 2: Exact DDC Token Choice Research

Goal:

- reproduce RADAN's compact DDC numeric token choices, not merely equivalent decoded geometry.

Deliverables:

- `generate_micro_dxf_corpus.py`
- `run_radan_micro_oracle.py`
- `analyze_micro_token_choices.py`
- `ddc_token_choice_model.json`

Micro corpus design:

- one horizontal line
- one vertical line
- one diagonal line
- one rectangle with known min bounds
- one circle
- one 90-degree arc
- one non-cardinal arc
- one tiny jog profile similar to `F54410-B-49`

Coordinate/value sweeps:

- exact dyadic values: `0.0625`, `0.125`, `0.25`, `0.5`
- decimal-looking values: `0.025`, `0.05`, `0.1`
- F54410 problem values: `0.510833`, `0.512607`, `6.299218`, `10.098437`
- values around rounding boundaries: `x.0000004`, `x.0000005`, `x.0000006`
- negative-source values where local min subtraction matters
- source min bounds rounded vs unrounded

Hypotheses to test:

- RADAN encodes source values before local min subtraction.
- RADAN rounds source coordinates and min bounds separately.
- RADAN keeps importer-internal double values from DXF parse, not displayed 6-decimal values.
- RADAN canonicalizes topology before encoding.
- RADAN token length depends on source decimal precision or DXF text representation.
- Line records and arc records use different value-selection rules.
- Arc records prefer center/radius/angle reconstruction over endpoint reconstruction.

Pass gate:

- reproduce 100 percent of micro-corpus DDC tokens for simple lines
- then reproduce circles
- then reproduce cardinal arcs
- then reproduce non-cardinal arcs

Decision point:

- if micro lines cannot be reproduced exactly, stop chasing full native generation and document why.

## Phase 3: Metadata and Cache Sensitivity

Goal:

- identify non-geometry fields that must be preserved or regenerated for RADAN to display and nest a symbol correctly.

Deliverables:

- `analyze_sym_metadata_fields.py`
- `mutate_sym_metadata_matrix.py`
- `metadata_sensitivity_matrix.json`

Mutation tests:

- change only DDC geometry token text while preserving decoded geometry
- change only bounds/extents
- change only record counts
- change part name/description attributes
- change pen fields only
- remove or zero suspected preview/cache fields
- preserve good DDC but mutate wrapper fields
- preserve wrapper but replace DDC with good DDC from another same-shape part

Observations to collect:

- file parser pass/fail
- DDC parser pass/fail
- decoded geometry pass/fail
- RADAN open status, if approved
- RADAN thumbnail/display status, if approved
- whether opening and saving in RADAN repairs the file
- whether nesting/print report uses stale geometry or live DDC

Pass gate:

- identify the minimum metadata set that must remain untouched
- identify any fields that RADAN regenerates on open/save
- identify whether a cache rebuild call exists or whether RADAN open/save is the only known rebuild

## Phase 4: Per-Part Donor Strategy

Goal:

- determine whether known-good per-part donors can make a safe narrow solution before full no-template generation is cracked.

Important distinction:

- Universal donor failed.
- Per-part known-good donor may still be useful.

Variants:

1. Exact preserve:
   - if a good `.sym` already exists for a part, preserve its DDC block and wrapper exactly.
   - only update project references and quantities.
2. Same-topology patch:
   - use the per-part good symbol wrapper/cache.
   - replace only DDC records whose source DXF entity count/type and topology match.
3. Same-family donor:
   - use a donor from a related part only if wrapper metadata proves non-specific.
   - this is lab-only until disproven safe.

Pass gate:

- no visible failures on known-bad canaries
- exact DDC and metadata fingerprints are explained
- RADAN open/nest/report is correct in a copied lab project

Promotion rule:

- per-part donor behavior can be considered for operations only if it never writes missing symbols from unrelated donors and never silently accepts topology mismatch.

## Phase 5: Controlled RADAN Oracle Run

Run this only after offline analysis has narrowed the questions.

Preflight:

1. enumerate RADAN/Radraft/Radnest/Radpunch processes
2. log PID, path, title
3. stop matching processes
4. re-check zero matching processes

Allowed RADAN uses:

- generate micro-corpus oracles from lab DXFs
- open a tiny hybrid matrix in RADAN for visual classification
- save a copied symbol to test whether RADAN regenerates cache fields

Disallowed:

- production project folders
- W-side writes
- broad full-project conversion
- visible user-session attachment
- hidden mutation of open user projects

Output:

- `radan_oracle_run_<timestamp>`
- process cleanup logs
- generated oracle syms
- RADAN session notes
- screenshots only if useful
- final process cleanup proof

Pass gate:

- RADAN produces deterministic oracle symbols across repeated runs from identical lab DXFs
- repeated oracle outputs either match byte-for-byte or have all variable fields classified

## Phase 6: Rebuild Writer Only After Evidence

Do not keep expanding the writer blindly.

Only update `write_native_sym_prototype.py` after the previous phases identify a concrete rule.

Candidate writer improvements:

- exact token-choice model for simple lines
- separate arc encoding rule
- metadata preservation map
- bounds/extents regeneration only for fields proven safe
- strict topology guard
- explicit provenance record in write report

Pass gate:

- `B-10` exact
- `B-16` exact or fully explained
- `B-14` exact or fully explained
- `B-17`, `B-27`, and `F54410-B-49` pass RADAN visual validation in a lab project
- arc canaries `B-28`, `B-30`, and `F54410-B-41` pass decoded and visual validation

## Acceptance Criteria

Research is useful if it produces one of these outcomes:

1. Full crack:
   - token and metadata model can reproduce RADAN-created symbols for target corpus.
2. Narrow safe crack:
   - per-part donor approach works safely for existing known-good symbols, with hard guards.
3. Negative proof:
   - evidence shows RADAN uses hidden/importer state or opaque cache fields we cannot regenerate safely.

Promotion remains blocked unless:

- output passes file-level validation
- output passes exact or explained token comparison
- output passes RADAN visual validation on known-bad canaries
- output passes copied-project nest/report validation
- the normal import path remains unaffected
